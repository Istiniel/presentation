JSON web tokens (JWT)

The concept of JSON Web Tokens and how to impement it

JSON Web Token (JWT) is an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information between parties

JWT structure

Header

Payload

Signature

Header

Service part, JWT "settings" (algorithm, token type)


					{
						"alg": "HS256",
						"typ": "JWT"
					}

Payload

Identifying information (user id, name etc)

here can be stored any useful data

					
					{
						"sub": "1234567890",
						"name": "Quincy Larson",
						"iat": 1516239022
					}

"iat" is the "issued at" date for the token, and is a registered claim

Signature

Cryptographic signature of the token

The header and payload parts are encrypted by base64url algorithm and afterwards united in a single box using a dot (‚Äú.‚Äù) as a divider 

					
						HMACSHA256(
							base64UrlEncode(header) + "." +
							base64UrlEncode(payload),
							secret-key
						)

Secret key is used to generate JWT signature. It has to be stored serverside and shared beetwen backend services

JWT advantages

JWT tokens are not stored on the server (optionally)

The backend verifies the signature and extracts necessary fields out of the payload, there is no reason to store tokens manually 

					
						const jwt = required('jsonwebtoken')

						jwt.verify(token, secretOrPublicKey, [options, callback])

With this approach, each service gets necessary user data directly from the token and does not waste time on acquiring this information from the database

Separated authentication server

JWT provides ability to use auth service separated from main server

Payload information is encoded

JWT is a string in the nutshell, divided by dots 

					
						header.payload.signature

Encoded payload provides ability to use cookie for data transfering

Encoding !== Encryption

JWT does not guaranty secure for sensitive information!

It's mainly the way sustain stateless approach for authorizaion process

Do not place passwords inside JWT payload!

Access and refresh tokens

Access token

Authorization server issues an access token, which is an artifact that client applications can use to make secure calls to an API server


Access tokens may be valid for a short amount of time

(75 minutes on average)

Refresh token

Lets a client application get new access tokens without having to ask the user to log in again

The client application can get a new access token as long as the refresh token is valid and unexpired

Refresh token that has a very long lifespan

Automatic reuse detection

Every time your application exchanges a refresh token to get a new access token, the authorization server also returns a new refresh-access token pair
